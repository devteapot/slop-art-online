# ADR 004: NPC Behavior Trees + Plans — Replacing FSM + Raw Tool Calls

**Status:** Implemented (server module compiles, bridge awaits binding regeneration)
**Date:** 2026-03-20

## Context

Phase 1 (tool calling) proved LLM-driven NPCs work end-to-end but exposed two problems:
1. Combat via tool calls has 2-4s LLM latency per decision — too slow for reactive combat
2. Tool calls can't express multi-step plans ("travel to market, sell sword, go home")

## Decision

Replace the FSM (`NpcBehaviourGraph`) and raw tool calls with a **two-layer behavior system**:

- **Layer 1 — Combat Tree (reactive):** Re-evaluated from root every 500ms tick. A default tree loads instantly on combat start (zero latency). The LLM generates an upgraded tree async and hot-swaps it when ready.
- **Layer 2 — NPC Plan (sequential):** Steps advance when completed. Interrupted by combat, resumed or replaced after. Generated by LLM when NPC is idle or post-combat.

Priority: Combat tree > Plan > Default wander.

Uses `bonsai-bt` crate for the `Behavior<A>` enum with serde support, but a custom stateless evaluator (not `BT::tick()`) since SpacetimeDB reducers are stateless.

## What Changed

### Server Module (`server/module/spacetimedb/`)

| File | Change |
|------|--------|
| `Cargo.toml` | Added `bonsai-bt = { version = "0.11", features = ["serde"] }` |
| `tables.rs` | Replaced `NpcBehaviourGraph` with `NpcBehavior` (mode + combat_tree JSON). Added `NpcPlan` (steps JSON + current_step). Added `decision_type` to `NpcPendingDecision`. |
| `npc_ai.rs` | Full rewrite. `NpcBtAction` enum (conditions + actions). `build_default_combat_tree()`. `evaluate_combat_tree()` stateless evaluator. `execute_bt_action()` maps actions to game effects. `execute_plan_step()` advances sequential plans. Helper functions for behavior upsert and decision triggering. Legacy `NpcAction` kept for `submit_npc_actions`. |
| `lib.rs` | Rewrote NPC loop in `tick_npcs` with two-layer priority. Added `submit_npc_combat_tree` and `submit_npc_plan` reducers. Removed `submit_npc_graph`. Updated `spawn_npc` to create `NpcBehavior`. |
| `combat.rs` | Updated `kill_npc` to clean up `npc_behavior` + `npc_plan` tables. |

### Bridge (`server/bridge/`)

| File | Change |
|------|--------|
| `prompt.rs` | Three prompt templates: combat tree, plan, post-combat. Each instructs the LLM to return structured JSON. |
| `llm.rs` | Replaced `generate_behaviour_graph()` and `generate_npc_actions()` with `generate_combat_tree()`, `generate_plan()`, `generate_post_combat()`. Shared `call_ollama()` helper. |
| `main.rs` | Routes decisions by `decision_type` field. Removed `BRIDGE_MODE` env var. Falls back to default tree / wander plan on LLM failure. |
| `tools.rs` | Kept but unused (`#[allow(unused)]`). May be repurposed for social tool calls later. |

### New Data Model

```
NpcBehavior { npc_id PK, mode: String, combat_tree: String }
NpcPlan { npc_id PK, steps: String, current_step: i32 }
NpcPendingDecision { npc_id PK, decision_type: String, context: String }
```

### Removed

- `NpcBehaviourGraph` table
- `submit_npc_graph` reducer
- `BRIDGE_MODE` env var
- Old FSM types (`BehaviourGraph`, `BehaviourNode`, `Transition`, `DEFAULT_GRAPH` in npc_ai.rs)
- Old graph evaluation (`evaluate_graph`, `check_condition`, `execute_action`)

### Kept

- `submit_npc_actions` reducer (testing, social one-off actions, escape hatch)
- `NpcDestination` table (used by plan's `travel_to`)
- `NpcChatMessage` table (used by `say` action)
- `tools.rs` (future social tool calls)

## What Remains

1. **Publish and regenerate bindings** — schema changed, requires `--delete-data`:
   ```bash
   just publish-reset
   just generate
   ```
2. **Rebuild bridge** — will compile after bindings include `submit_npc_combat_tree`, `submit_npc_plan`, and the updated `NpcPendingDecision` type:
   ```bash
   cargo build --package bridge
   ```
3. **End-to-end testing:**
   - Spawn NPC, walk near it -> combat_start -> default tree runs immediately -> LLM upgrades tree
   - Walk away -> post_combat -> LLM generates plan -> NPC follows plan steps
   - Walk back mid-plan -> combat tree takes over -> plan pauses -> combat ends -> LLM decides next
4. **Client bindings** — client may reference old `NpcBehaviourGraph` table; check and update if needed after `just generate`.

## Future

- More plan actions: trade, buy, sell, pick_up, equip (Phase 2 NPC tables)
- `build_default_combat_tree()` generates trees from NPC skill set (Phase 2)
- Combat tree re-generation on significant changes (health threshold, reinforcements)
- Idle plan generation (trigger LLM when NPC has nothing to do)
- Concurrent bridge processing (tokio::spawn per decision)
