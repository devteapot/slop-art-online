# ADR 005: NPC Architecture v2 — Identity, Emotion, Unified Trees

## Status: Accepted (2026-03-26)

## Context

The initial NPC AI system (ADR 003) had separate combat/life trees, a mode-switching tick loop (`sleeping | combat | plan | life_tree | idle`), and no structured personality or emotion system. The `persona` field was a flat string. Identity only updated at sleep (day→night transition). The LLM was called for every combat start, every social interaction, and every plan — too expensive for the vision of thousands of NPCs.

This ADR captures the redesigned architecture based on the project's vision: **NPCs are living entities that evolve through experience**. They learn, form opinions, feel emotions, and become more capable over time. A wolf that survives many fights might eventually become something more interesting than a wolf. The LLM is a creative consultant called sparingly, not a real-time brain.

## Design Principles

1. **The behavior tree is the NPC's habits and instincts** — it handles 95% of situations without any LLM call
2. **The LLM is a creative consultant** — called sparingly for tree generation and novel content
3. **Identity evolves through experience** — both through cheap inline BT actions and rare LLM evaluations
4. **Mobs and NPCs are on the same spectrum** — a wolf is just an NPC with a simple tree and minimal identity
5. **Emotions are the runtime modifier** — same tree, different emotional state = different behavior
6. **Knowledge expands capability** — what an NPC knows determines what actions are available to it

## Decisions

### 1. Unified Behavior Tree

**What:** Replace `NpcBehavior { mode, combat_tree, life_tree }` and `NpcPlan` with a single `current_tree` field.

**Why we moved away from mode switching:** The v1 tick loop was a big if/else chain:
```
sleeping? → regen + walk home
combat?   → evaluate combat_tree
plan?     → execute plan step
life_tree? → evaluate life_tree
idle?     → wander
```
This created sync bugs: what happens when an NPC is in `life_tree` mode but gets attacked? It needs to swap to `combat` mode, load the combat tree, etc. What if a conversation starts during combat? The mode switching was fighting the behavior tree's natural strengths.

**How it works:** The tree handles all situations through priority layers:

1. **Reactive Layer** — combat response, conversation protocol (always active, highest priority)
2. **Awareness Layer** — threat evaluation, social cues, enemy assessment
3. **Daily Life Layer** — goal pursuit, tasks, exploration
4. **Fallback Layer** — wander, rest

No mode switching. If an enemy appears, the reactive layer fires via conditions. When the enemy leaves, conditions no longer match and the tree falls through to daily life. The LLM generates complete trees covering all anticipated situations.

**Why plans become tree nodes:** `NpcPlan` was a separate execution path (step-by-step list) that bypassed the tree. This meant two evaluation systems. Plans are now expressed as `Sequence` nodes in the tree, keeping one execution model.

#### Example: Enemy Encounter (Cautious Guard NPC)

This tree is generated by the LLM at dawn. Personality shapes the structure — a guard gets observe-first branches:

```
Select [
  // REACTIVE (always checked first)
  Sequence [IsBeingAttacked, CombatResponse]
  Sequence [BeingAddressedInConversation, ConversationProtocol]

  // AWARENESS (perceive threats without acting)
  Sequence [
    EnemyDetected,
    NOT(IsBeingAttacked),
    EvaluateStrength,                          // sets internal belief
    Select [
      // Mission requires eliminating this enemy type, and I'm not too scared
      Sequence [
        GoalRequires("eliminate", target_type),
        EmotionBelow("fear", 0.6),
        AttackEntity { entity_type: Npc, entity_id: 7 }
      ],
      // I'm much stronger + angry → intimidate
      Sequence [
        StrengthAdvantage(0.5),
        EmotionAbove("anger", 0.4),
        SayToEntity { entity_type: Npc, entity_id: 7, message: "Move along." },
        KeepDistance(8.0)
      ],
      // Default: note presence, continue my day
      Sequence [
        SetBelief { subject: "entity:npc:7", predicate: "spotted_threat", object: "true" },
        KeepDistance(12.0)
      ]
    ]
  ]

  // DAILY LIFE
  Sequence [TravelToEntity(Poi, 3), PatrolArea]

  // FALLBACK
  Wander
]
```

#### Same Scenario: Bandit NPC

The LLM generates a fundamentally different tree because the bandit's personality is aggressive:

```
Select [
  Sequence [IsBeingAttacked, CombatResponse]
  Sequence [BeingAddressedInConversation, ConversationProtocol]

  Sequence [
    EnemyDetected,
    NOT(IsBeingAttacked),
    EvaluateStrength,
    Select [
      // Attack if I think I can win (even slightly weaker is OK for a bandit)
      Sequence [StrengthAdvantage(-0.2), EmotionBelow("fear", 0.8), Attack]
      // Stalk if much weaker
      Sequence [StrengthAdvantage(-0.5), Follow(distance: 20.0), WaitForOpportunity]
      // Outmatched: flee
      Flee
    ]
  ]

  Sequence [LootArea]
  Wander
]
```

**Key insight:** Personality doesn't need runtime conditions for everything — it's **baked into the tree structure** at generation time. The guard's tree has "observe and ignore" as default. The bandit's tree has "attack if possible" as default. Emotions add the runtime twist: the normally-calm guard, angered by a recent ally death, activates the intimidation branch.

#### The "Self-Adapting" Tree

The tree doesn't literally rearrange — but its evaluation path changes constantly based on runtime state. The same tree produces different behavior every tick because:
- Emotions change (events trigger anger/fear/joy, then decay)
- Beliefs change (new information from listening, gossip)
- Knowledge changes (learned new mechanics)
- World state changes (day/night, nearby entities, health)

This means the tree stays relevant without regeneration for most situations.

### 2. Tree Regeneration Triggers (exhaustive)

**Why be exhaustive about this:** Every regen trigger is an LLM call. The wrong triggers = too expensive. The right triggers = NPCs feel alive without breaking the budget.

The LLM generates a new tree ONLY in these cases:

**Scheduled:**
- Dawn wake-up (new day, plan the day)
- Post-reflection (significant identity shift after experience evaluation)

**Event-driven:**
- Tree exhaustion (N consecutive ticks with no action produced — the NPC has literally nothing left to do)
- Goal completed or failed (the tree was built to pursue a goal that just resolved — it's obsolete)
- Major knowledge gain (first knowledge in a *new category* opens entirely new action types the tree can't have anticipated)
- Near-death survival (survived at <10% health — current approach almost killed the NPC, needs rethinking)

**Self-requested:**
- `RequestNewTree` BT action placed by the LLM at the end of a sequence ("after finishing this trade run, reassess")

**NOT triggers (handled by existing tree conditions):**

| Situation | How the existing tree handles it |
|---|---|
| New enemy appears | Reactive layer: `IsBeingAttacked` or awareness layer conditions |
| Someone talks to me | Reactive layer: conversation protocol interrupt |
| Day/night transition | `IsNightTime`/`IsDayTime` conditions |
| Emotion spike | `EmotionAbove`/`EmotionBelow` conditions |
| Enemy leaves | `EnemyDetected` becomes false, tree falls through to daily life |
| Health drops | `HealthBelow` conditions |
| Item picked up | Inventory conditions update |

**Decision rationale:** We initially considered triggering tree regen on each new enemy encounter or each conversation. This would have been ~10-50 LLM calls per NPC per day. The exhaustive trigger list keeps it to ~1-3 calls per common NPC per day cycle, zero for mobs.

### 3. NPC Identity Model

Six components, all persisted:

- **Personality** — Structured traits (aggression, sociability, curiosity, courage, empathy, discipline). Evolves slowly through LLM-evaluated significant experiences. Defines emotion baselines and decay rates.
- **Beliefs** — Subjective assessments about entities/situations. Confidence 0.0-1.0. Can be wrong. Spread through gossip.
- **Knowledge** — Learned world mechanics and facts. Expands the NPC's action space for tree generation. Shareable.
- **Goals** — Long-term desires with priority levels and success conditions.
- **Relations** — Disposition toward specific entities (-100 to +100) with context.
- **Emotions** — Runtime state (anger, fear, joy, sadness, surprise, disgust). Changes every tick. Decays toward personality baselines.

**Why we didn't add a separate Mission table:** We considered separating Goal (long-term) from Mission (short-term). Decision: the current behavior tree IS the mission. It's generated with goals in mind. Adding a separate mission table risks desync between "what the mission says" and "what the tree actually does." If we later need mission-level tracking (progress, deadlines, multi-session persistence), we can add it then.

### 4. Knowledge vs Beliefs vs Memory

| Type | What | Example | Decays? | Key use |
|---|---|---|---|---|
| Memory (event log) | Things that happened | "I fought a wolf near the gate" | Yes (5min TTL) | Short-term context for reflection |
| Beliefs | Subjective assessments | "Player Alice is dangerous" | No (updated/replaced) | Social decisions, gossip propagation |
| Knowledge | Learned mechanics/facts | "Potions cost ~10g at market" | No (updated/replaced) | **Expands BT action space** |

**Why knowledge is separate from beliefs:** Initially we had only beliefs. But beliefs are opinions ("the forest is dangerous") while knowledge is mechanics ("wolves are weak to fire"). The critical difference: knowledge determines what actions an NPC can take. An NPC that doesn't *know* about potions can't have a "buy potion" branch in its tree. Beliefs influence decisions within existing branches; knowledge creates entirely new branches.

**Knowledge is also the most impactful thing to share between NPCs.** A veteran guard teaching a recruit "aim for the legs" gives the recruit new combat capabilities, not just an opinion.

### 5. Knowledge-Gated Entity References

**Why this matters:** In v1, BT actions used loose strings (`TravelToPoi("Market Square")`). The LLM already receives concrete entity data in its context (NPC IDs, POI IDs, item_def_ids). Actions should reference real entities for deterministic execution.

**But new NPCs don't know about everything.** A freshly spawned NPC shouldn't reference the market by ID if it's never been there. The NPC's knowledge gates what the LLM can reference.

BT actions come in two forms:

**Vague** (no knowledge required):
- `SearchFor("healing")` — wander looking for something matching the category
- `ExploreArea` — unguided exploration
- `Investigate("toward noise")` — directional exploration

**Concrete** (knowledge required, validated against NpcKnowledge):
- `TravelToEntity(Poi, 3)` — go to a known location
- `BuyItem { from_npc_id: 5, item_def_id: 12, max_price: 15 }` — specific transaction
- `UseItem { item_def_id: 12 }` — use a known item

The LLM prompt constrains: "Only generate concrete entity references for things in the NPC's knowledge."

#### Example: Knowledge Progression

**Stage 1 — No knowledge (NPC is hurt, never seen healing):**
```
Sequence [HealthBelow(0.5), Rest]    // Only option: sit and regen slowly
```

**Stage 2 — Vague knowledge (overheard someone mention "potions heal you"):**
```
Sequence [HealthBelow(0.5), SearchFor("healing")]
// NPC wanders looking for anything healing-related
// If it finds a potion → AddKnowledge fires, learns item_def_id + location
```

**Stage 3 — Concrete knowledge (has seen/used a potion, knows the market sells them):**
```
Sequence [
  HealthBelow(0.5),
  TravelToEntity(Poi, 3),                              // Market Square
  BuyItem { from_npc_id: 5, item_def_id: 12, max_price: 15 },
  UseItem { item_def_id: 12 }
]
```

**Validation happens twice:** At tree submission (entity exists?) and at execution (still exists? — world may have changed). Stale references return `Failure` and the tree falls through.

### 6. Emotion System

**Why emotions, not just personality:** Personality alone can't express "the normally calm guard is furious because an ally just died." You need a fast-changing layer on top of slow-changing personality. Emotions are that layer.

Emotions are the runtime modifier that makes identical trees produce different behavior.

#### Event Triggers (deterministic rules, no LLM)

| Event | Emotion Change |
|---|---|
| Took damage | anger +0.3, fear +0.2 |
| Won fight | joy +0.3, fear -0.2 |
| Was insulted | anger +0.4, sadness +0.1 |
| Was betrayed | anger +0.5, disgust +0.4, sadness +0.3 |
| Received gift | joy +0.2 |
| Near-death experience | fear +0.5 |
| Achieved a goal | joy +0.4 |
| Ally died nearby | sadness +0.5, anger +0.2 |
| Witnessed beauty/wonder | joy +0.2, surprise +0.3 |

These are applied immediately in the tick reducer when the event occurs. Cheap and deterministic.

#### Decay Toward Personality Baseline

```
emotion = lerp(emotion, baseline, decay_rate)
```

Where:
- `anger_baseline = personality.aggression` (aggressive NPC rests at higher anger)
- `fear_baseline = 1.0 - personality.courage` (cowardly NPC rests at higher fear)
- `joy_baseline = personality.sociability` (sociable NPC rests at higher joy)
- `decay_rate` is influenced by `personality.discipline` (high discipline = fast decay = stoic)

An emotionally volatile NPC has slow decay — emotions linger. A disciplined monk has fast decay — quickly returns to baseline.

#### Example: Same Event, Different NPCs

Being insulted by a player (anger +0.4):

| NPC | Personality | After insult | What happens |
|---|---|---|---|
| **Warrior** | aggression 0.8, courage 0.9 | anger spikes to 0.9 | `EmotionAbove("anger", 0.7)` triggers → attacks immediately |
| **Monk** | discipline 0.9, aggression 0.1 | anger rises to 0.3, decays fast | Back to baseline in a few ticks, ignores insult |
| **Merchant** | sociability 0.8, courage 0.3 | fear rises to 0.5, sadness 0.3 | `EmotionAbove("fear", 0.4)` triggers → ends conversation, retreats |

Same tree structure could be used, but the emotion gates route each NPC to a different branch. The warrior's tree has an anger-gated attack branch. The merchant's tree has a fear-gated retreat branch. The monk's tree barely has emotion gates at all — discipline overrides.

#### Emotions Affect Visuals (client-side, free)

The client reads the `NpcEmotion` table and shows:
- Angry NPC → walks faster, stomps, red tint
- Sad NPC → walks slowly, looks down
- Joyful NPC → bouncy walk, waves at passersby
- Fearful NPC → looks around nervously, flinches

No LLM needed. Pure visual feedback from table data.

### 7. Conversation Protocol (within BT, minimal LLM)

**Why not LLM-per-turn:** Having the LLM generate every conversation response would be ~50-100 calls per NPC per day. The conversation protocol handles the *flow* of conversation in the BT. The LLM only generates *content* when templates and knowledge can't handle it.

**Key architectural insight: separate listening from responding.** These are two different operations. You don't lose information by giving a template response — because listening already captured it.

#### Listening (always happens, zero LLM cost)

Every speech event near the NPC is logged with full detail:
```json
{
  "type": "heard_speech",
  "time_us": 1679832000000000,
  "location": { "x": 45.2, "z": 82.1, "poi": "Market Square" },
  "speaker": { "type": "player", "id": "abc123", "name": "Alice" },
  "target": { "type": "npc", "id": 5, "name": "Blacksmith" },
  "content": "I heard bandits are hiding in the eastern cave",
  "topic": "bandits",
  "engagement": "overhearing",
  "confidence": 0.2
}
```

Beliefs/knowledge created from listening inherit the engagement-scaled confidence.

#### Engagement Levels

| Level | Condition | Confidence multiplier |
|---|---|---|
| **Focused** | Active conversation with the speaker | 1.0 |
| **Attentive** | Nearby and idle/low-priority task | 0.5 |
| **Overhearing** | Nearby but busy (in combat, trading, focused task) | 0.2 |
| **Distant** | Edge of hearing range | 0.1 |

Additional modifier: `topic_relevance` (1.0 if matches NPC role/goals, 0.5 otherwise).

```
effective_confidence = base_confidence * engagement * topic_relevance
```

**Why topic relevance matters:** A guard and a merchant standing in the same square overhear the same conversation about bandits. The guard absorbs it better (topic_relevance 1.0, it's their job) than the merchant (topic_relevance 0.5, not their concern). Same engagement, different absorption.

#### Worked Example

Player A tells Player B: "The dragon burned down the eastern village."

| NPC | Engagement | Topic Relevance | Confidence |
|---|---|---|---|
| Guard (in active convo with speaker) | 1.0 (focused) | 1.0 (threats are their job) | **1.0** |
| Guard (patrolling nearby) | 0.5 (attentive) | 1.0 (threats) | **0.5** |
| Blacksmith (hammering at forge) | 0.2 (overhearing) | 0.5 (not their domain) | **0.1** |
| Merchant (at edge of square) | 0.1 (distant) | 0.5 (not their domain) | **0.05** |

The focused guard gets a high-confidence belief: "dragon burned eastern village (confidence 1.0)." The merchant barely registers it: "something about a dragon? (confidence 0.05)." This prevents eavesdropping from being overpowered while still allowing information to flow.

**Why the penalty for not being locked in matters:** Without engagement scaling, an NPC stationed in the town square all day would absorb hundreds of conversations at full confidence. It would know everything about everyone. That's overpowered and unrealistic. The engagement penalty means only focused interactions produce high-confidence knowledge.

#### Responding (tiered, mostly cached)

```
Select [
  // Tier 1: Canned responses (~60% of interactions)
  Sequence [MatchesTemplate("greeting"), SayTemplate("greeting_response")]
  Sequence [MatchesTemplate("farewell"), SayTemplate("farewell_response")]
  Sequence [MatchesTemplate("directions"), SayFromKnowledge("navigation")]
  Sequence [IsTradeRequest, StartTradeProtocol]

  // Tier 2: Knowledge-based (~25%)
  Sequence [TopicMatchesKnowledge, SayFromKnowledge(topic)]
  Sequence [TopicMatchesBelief, SayFromBelief(topic)]

  // Tier 3: Personality-colored template (~10%)
  Sequence [EmotionDominant("fear"), SayTemplate("nervous_deflection")]
  Sequence [EmotionDominant("joy"), SayTemplate("cheerful_ignorance")]

  // Tier 4: LLM generation (~5% — truly novel/important only)
  Sequence [IsImportantConversation, GenerateLlmResponse(context), SayGenerated]

  // Tier 5: Fallback
  SayTemplate("generic_acknowledgment")     // "Hmm, I see."
]
```

**The `IsImportantConversation` gate** controls LLM cost. It checks:
- Is this a high-relationship speaker (friend/enemy)?
- Has this conversation gone more than 3 exchanges? (player is invested)
- Is the topic something with no matching knowledge/belief? (need LLM to improvise)
- Is the NPC a key NPC (always use LLM for conversations)?

For a generic town guard, most conversations never reach Tier 4. For the main quest-giver, `IsImportantConversation` is almost always true.

**Why we don't lose information with template responses:** Listening happens first, always. Even if the NPC responds with "Hmm, I see" — it already logged "player mentioned dragon attack" as a belief and event. The NPC's identity evolves from the listening, not the response quality. At the next dawn reflection, the LLM sees this belief and can generate a tree that reacts to it.

**NPC-to-NPC conversations** work the same protocol but are simpler — they can be fully template/knowledge-driven since no human expects novel responses. NPC A's tree has a "share knowledge with nearby friend" branch → triggers NPC B's conversation protocol → knowledge transfers → both resume.

### 8. Identity Updates

**Design decision: identity updates as "subconscious" side effects, not a separate engine.**

We considered a dedicated "Identity Engine" as a separate system from the behavior tree. Decision: integrate identity updates directly into the BT as inline actions. This is like subconscious processing — the NPC's identity shifts as a byproduct of living, not from deliberate self-reflection (most of the time).

Two mechanisms:

**Inline BT actions** (cheap, during normal execution):
- `SetBelief`, `AddKnowledge`, `AdjustRelationship`, `TriggerEmotion`
- Placed in the tree by the LLM where contextually appropriate
- No LLM call, just DB writes

Example: after a combat action, the tree might include:
```
Sequence [
  Attack,
  SetBelief { subject: "combat", predicate: "experienced", object: "true" },
  AdjustRelationship { target: enemy, delta: -10 },
  AddKnowledge { category: "combat", fact: "wolves attack in packs" }
]
```

The NPC fights AND its identity shifts as a byproduct. No pause, no LLM call.

**Async experience evaluation** (rare, after significant events):
- Significant event triggers `NpcPendingDecision` with type `"experience"`
- LLM evaluates the experience and returns identity deltas:
  ```json
  {
    "personality_deltas": { "courage": +0.05, "empathy": -0.02 },
    "beliefs": [{"subject": "player:abc", "predicate": "betrayer", "object": "true"}],
    "knowledge": [{"category": "social", "fact": "player:abc cannot be trusted in trade"}],
    "relationship_updates": [{"target": "player:abc", "delta": -30}],
    "emotion_adjustments": { "anger": +0.3, "disgust": +0.2 }
  }
  ```
- Applied in background, NPC keeps acting on current tree
- If identity shift is large enough, triggers tree regeneration

**Why LLM for personality updates instead of rules:** We considered rule-based personality evolution ("if NPC nearly dies, increase caution by 0.1"). But what if the NPC nearly dies *saving a friend*? That might increase bravery, not caution. The meaning of an experience depends on context, and that's what LLMs are good at. The LLM interprets meaning; the reducer validates bounds.

### 9. Belief & Knowledge Propagation

Pure reducer logic, no LLM needed:

```
NPC A is near NPC B (within chat range)
NPC A has belief/knowledge with confidence > 0.7
NPC A's trust toward B > 30
→ B receives the belief/knowledge at reduced confidence
  (scaled by A's trust: belief.confidence * trust/100)
```

**Propagation chain degradation:**
- Merchant (confidence 0.9) tells Innkeeper (trust 80): Innkeeper gets confidence 0.72
- Innkeeper (confidence 0.72) tells Guard (trust 60): Guard gets confidence 0.43
- Guard (confidence 0.43) tells Recruit (trust 50): Recruit gets confidence 0.22

Information degrades as it spreads. High-confidence facts propagate far. Rumors fade.

**Player influence:** When a player tells an NPC something in chat, the social response can include `AddKnowledge` or `SetBelief`. "That merchant sells poison" → the NPC gets that belief (confidence based on engagement + relationship disposition), which it might propagate to friends.

**Propagation runs periodically** (every few ticks, not every tick) and only between NPCs within chat range. This naturally creates gossip networks — information spreads geographically and through trust chains.

### 10. Cost Model

| NPC Tier | Count | LLM Usage | Cost |
|---|---|---|---|
| **Mobs** (wolves, slimes) | Thousands | Static default trees, no LLM | Zero |
| **Common NPCs** (guards, merchants) | Hundreds | Tree at dawn + rare events | ~2-5 calls/day each |
| **Key NPCs** (quest givers, bosses) | Dozens | Trees + novel conversations | ~10-30 calls/day each |

**The mob → NPC spectrum:** A basic wolf's tree is never LLM-generated:
```
Select [
  Sequence [IsBeingAttacked, Attack]
  Sequence [EnemyDetected, StrengthAdvantage(-0.3), Attack]
  Sequence [EnemyDetected, Flee]
  Sequence [IsNightTime, Wander]
  Rest
]
```

Zero identity, zero LLM cost. But if that wolf survives many fights, gains knowledge ("players with shields are hard to kill"), and hits a rare significance threshold... the LLM could generate a smarter tree for that specific wolf. A wolf that *learned* becomes more interesting than its kin. This is emergent — you don't script it. The significance threshold is just set very high for mobs.

## Consequences

- Simpler tick loop (no mode switching, one tree evaluation per tick)
- NPCs naturally learn and evolve through experience
- LLM costs are controlled and predictable
- Mobs and NPCs are on the same spectrum (simple → complex)
- Social interactions are mostly free (template/knowledge-driven)
- Emotions provide runtime behavioral variation without LLM calls
- Knowledge-gating creates natural learning curves
- Information degrades through gossip chains (realistic, prevents omniscience)
